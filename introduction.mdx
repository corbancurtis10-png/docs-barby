EZAnimation is a Roblox Luau ModuleScript that takes the complexity out of working with the Roblox `AnimationController` and `Animator` APIs. Instead of wiring up tracks, managing priorities, and handling cleanup manually, you initialize a single service, pass it your animation IDs, and call straightforward methods like `play`, `stop`, `crossFade`, and `queue` — all without touching the underlying Roblox animation plumbing.

## Why EZAnimation?

Roblox's built-in animation system requires you to load each `Animation` instance onto an `Animator`, store the returned `AnimationTrack`, manage its lifecycle, and remember to disconnect events when you're done. On any project with more than a handful of animations this becomes tedious and error-prone. EZAnimation solves these problems by:

- **Centralizing your animation library** — define all your asset IDs in one place at initialization.
- **Abstracting track management** — the service loads, caches, and cleans up tracks for you.
- **Providing higher-level primitives** — queuing, cross-fading, grouping, state machines, ragdoll, and tween helpers ship out of the box.
- **Keeping the API consistent** — every method accepts the same `model` and string `id` pattern, so there's nothing new to learn as your project grows.

## How It Works

EZAnimation ships as a single Roblox **ModuleScript**. You place it in `ReplicatedStorage` (or any other location accessible to your scripts), `require` it, and call `AnimationService.init()` with a table mapping string keys to `rbxassetid://` strings. From that point on, every method call references your animations by those string keys — no `Instance.new("Animation")` boilerplate required.

```lua
local AnimationService = require(game.ReplicatedStorage.EZAnimation)

local anim = AnimationService.init({
    List = {
        idle    = "rbxassetid://1234567890",
        run     = "rbxassetid://0987654321",
        attack  = "rbxassetid://1122334455",
    }
})

anim:play(character, "idle", "Idle", true)
```
